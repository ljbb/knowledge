A. 全文搜索引擎升级

  当前用 SequenceMatcher 做字符串匹配。知识库超过 50 条后精度下降明显。接入 qmd（llm-wiki 推荐的 BM25 + 向量搜索 + LLM 重排序），或者用 sqlite-vec/chromadb 做本地向量搜索。

B. Web 端直接导入（不依赖 LLM）

  当前上传 → LLM 整理 → 入库。可以加一条路径：上传时如果格式已经符合模板，Engine 直接解析 frontmatter + Markdown 入库，跳过 LLM 步骤。LLM 不可用时也能导入。

C. 知识版本 Diff 视图

  raw add-only + 版本升级已经有了，但 Web 上只能看到最新版本。加一个版本历史页，展示 v1 → v2 → v3 的变更对比。

 扩展方向 /高成本

D. 知识图谱可视化

  用 D3.js 或 vis.js 渲染知识之间的 [[关联知识]] 链接关系，在 Web 首页展示一个交互式的节点图。Obsidian 的 Graph View 就是这个概念。

E. 定时巡检 + 通知

  kb.py check 做成 cron job，发现过期/孤立知识时通过 Slack Webhook 或邮件通知。

F. 知识导出（PDF / 离线 HTML / Marp 幻灯片）

  某个分类的所有知识一键打包成 PDF 手册或 Marp 演示文稿，方便离线查阅或分享给没有访问权限的人。

  共享服务版需要的改动

  ┌──────────┬────────────────────┬────────────────────────────┐
  │   模块   │        当前        │            改为            │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 存储     │ Markdown 文件      │ PostgreSQL（knowledge 表） │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 搜索     │ SequenceMatcher    │ pgvector 语义搜索          │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 并发     │ 文件锁             │ PG 事务 + MVCC             │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 知识本体 │ wiki/*.md          │ DB 主存储 + Markdown 导出  │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 部署     │ python kb.py serve │ Docker Compose（app + PG） │
  ├──────────┼────────────────────┼────────────────────────────┤
  │ 认证     │ 无                 │ 至少 Basic Auth / API Key  │
  └──────────┴────────────────────┴────────────────────────────┘

  最大的改动只有一个：存储层从文件系统换成 PostgreSQL。 Engine 其他模块（ingest → filter → quality → skill_gen）逻辑不变，只是读写从 open("wiki/...") 变成 SQL。

  Markdown 文件可以保留作为导出格式 — 每录入一条知识，同时生成 .md 和 .html，方便 LLM 直接读文件、git 备份、或者离线查看。

● LLM 不碰数据库，走的还是现有路径，存储层对 LLM 完全透明。

  ---
  LLM 访问知识的路径（不变）

  LLM (Claude Code / Copilot / OpenCode)
      │
      ├── kb-search skill ──▶ python kb.py search "..." ──▶ Engine ──▶ PG ──▶ JSON 结果
      │
      ├── kb-ingest skill ──▶ LLM整理内容 ──▶ python kb.py ingest ──▶ Engine ──▶ PG
      │
      └── REST API ──▶ GET /api/search?q=... ──▶ Engine ──▶ PG ──▶ JSON 结果

  LLM 永远只看到：
  - Skill 文件（不变的 agentskill.io 标准格式）
  - CLI 命令（python kb.py search，接口不变）
  - REST API 返回的 JSON（格式不变）

  SQLite、PostgreSQL 还是 Markdown 文件，LLM 完全不知道也不关心。

  ---
  关键设计：Markdown 同步导出

  一条知识入库时，同时写两个地方：

  kb-ingest
      │
      ├──▶ PostgreSQL (主存储，搜索/查询/并发)
      │     knowledge 表: id, title, category, content, embedding, ...
      │
      └──▶ wiki/*.md + wiki/*.html (自动同步导出)
            ├── 给 Copilot 直接读文件
            ├── git 版本管理
            └── 人类离线浏览

  ┌────────────────────┬────────────────────────┬────────────────────┐
  │      访问方式      │         读的是         │      适用场景      │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ kb-search (LLM)    │ PostgreSQL（向量搜索） │ 语义搜索，秒级定位 │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ kb-search (CLI)    │ PostgreSQL（向量搜索） │ 命令行快速查       │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ Web /api/search    │ PostgreSQL（向量搜索） │ 浏览器搜索         │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ Copilot 直接读文件 │ wiki/*.md              │ LLM 零依赖访问     │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ 浏览器阅读         │ wiki/*.html            │ 人类离线查看       │
  ├────────────────────┼────────────────────────┼────────────────────┤
  │ git 版本管理       │ wiki/*.md              │ 备份、diff、审计   │
  └────────────────────┴────────────────────────┴────────────────────┘

  PG 是主存储，Markdown 文件是自动同步的镜像。 两边内容一致，谁都不丢。

  ---
  新增能力：MCP Server

  有了 PG 做后端，加 MCP Server 就很简单 — Engine 已经有完整的 API，包一层 MCP 协议即可：

  LLM ──MCP──▶ kb_search("nginx 重启")
              ▶ kb_get("kb-2026-0711-001")
              ▶ kb_ingest(source_text="...")
              ▶ kb_check()

  不再需要通过 shell 调 CLI，LLM 原生调用知识库工具。这对 Claude Code 体验提升很大。
